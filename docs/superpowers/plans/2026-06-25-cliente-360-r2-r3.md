## Tareas

### Backend (msp-api)

**T1 — R3.1 dominio: posteriors bayesianos del BG/NBD**
- Nuevo `internal/analytics/app/btyd_posteriors.go`: a partir de los params BG/BB + Gamma-Gamma ya
  embebidos, generar N draws del posterior por cliente → intervalos creíbles (p.ej. 90%) para
  `PAlive`, `ExpectedPurchases(12m)`, y `CLV`. Determinístico (seed fija derivada del cliente_id; NO
  usar Math.random global — el harness Go lo permite con `math/rand` sembrado). Función pura.
- **"Próxima compra estimada":** derivar de recencia + cadencia de recompra inferida (meses distintos
  de venta / ventana) con un rango. Pura.
- Tests (≥99% dominio, property con `pgregory.net/rapid`, fuzz): el intervalo contiene el estimador
  puntual; ancho del intervalo decrece con más historia; monotonicidades; sin panics.

**T2 — R3.1 contrato + endpoint de predicciones**
- `PrediccionesClienteContract` en analytics (puntual + IC por métrica + próxima compra estimada) +
  `analyticsapp.Service.ObtenerPredicciones(clienteID)`.
- Puerto outbound en clientes (`PrediccionesClient`) + adapter en `cmd/api/clientes_wiring.go`.
- Endpoint `GET /v2/clientes/{id}/predicciones` (auth `PermClientesLeer`), DTO con strings para
  números monetarios, IC como `[lo, hi]`. Degradación: si no hay historial suficiente → 200 con
  `disponible:false`.
- Tests app (fakes) + handler.

**T3 — R3.2 peer benchmark (percentil vs cohorte)**
- `internal/analytics/app/percentile.go`: dado el valor del cliente y un cohorte (zona|segmento|todos),
  calcular percentil + media + p25/p50/p75 del cohorte sobre `MSP_AN_WINBACK_CANDIDATOS`. v1
  **on-the-fly** (una query agregada por cohorte; medir; materializar después si hace falta).
- Métricas benchmarkeables: score winback, score_credito, score_recompra, CLV, pct_pagos_a_tiempo.
- Endpoint `GET /v2/clientes/{id}/benchmark?cohort_by=zona|segmento` → por métrica: valor, percentil,
  cohorte stats. Contract + puerto + adapter.
- Tests app (percentil correcto en distribución conocida) + query integración (rollback) + handler.

**T4 — R3.4 tendencia sobre las series**
- `internal/analytics/app/trend.go` (o en clientes según dónde vivan las series): pendiente (regresión
  lineal simple) + bandera de cambio (último vs media previa, umbral) sobre `AbonosPorMes` y
  `CompradoVsAbonado`. Pura.
- Exponer en `SeriesDTO` (o nuevo bloque) de la ficha: `tendencia{slope, direccion, cambio}`.
- Tests (serie creciente → slope>0; plana → 0; cambio detectado).

**T5 — R3.3 timeline unificado (compras + pagos)**
- `GET /v2/clientes/{id}/timeline`: feed ordenado por fecha mezclando eventos de compra (DOCTOS_PV) y
  pago (MSP_PAGOS_VENTAS / ritmo). Reusar las lecturas existentes de ventas + ritmo; un solo DTO
  `EventoTimeline{fecha, tipo, monto, etiqueta, refId}`. Tipos: compra_credito, compra_contado,
  pago, liquidacion. (Visitas: campo `tipo` extensible a futuro.)
- Tests app (merge ordenado, tipos) + handler.

### Frontend (sistema-cobro-web)

**T6 — R2 refactor a tabs (demo-first, primero)**
- `ClienteFicha.tsx` → `Tabs` (shadcn) con 5 tabs reusando componentes existentes:
  - **Resumen:** Hero + SaludStrip + KPIs + NBA + (sparkline mini) + lectura analista (resumen corto).
  - **Análisis & predicción:** (placeholders que T7–T9 llenan) — al inicio muestra los scores (mover
    `FichaInteligenciaScores` aquí).
  - **Pagos & solvencia:** `FichaRitmoPago` + cadencia/puntualidad.
  - **Productos:** `FichaVentasList` + next-best-product.

**T7 — R3.1 FE: predicciones con IC**
- `FichaPredicciones.tsx` (en tab Análisis): 3 tarjetas (P(alive) como gauge de confianza; próxima
  compra estimada con rango; CLV con banda). Consume `useDesglosePredicciones(clienteId)` (hook + port
  + adapter + dto + mapper). Visual de IC (barra con whiskers / banda). Tests vitest + hook.

**T8 — R3.2 FE: peer benchmark**
- `FichaBenchmark.tsx`: gauge de percentil por métrica + mini distribución del cohorte; selector

**T9 — R3.3 FE: timeline unificado**
